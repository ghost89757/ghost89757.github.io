1 进程同步与信号量

1.1 多进程合作问题

想象一下，一群人去坐大巴。大巴上有司机和售票员，司机负责大巴的操作，售票员负责开关门和售票，正常情况下，售票员开门售票，售完票关门，司机再发车。司机和售票员之间存在合作，才能保证大巴的合理运行。设想，司机和售票员没有合作，那么很可能，当售票员正开门售票时，大巴突然就向前行驶了。

司机和售票员分别就是一个进程，他们合作完成一个任务：大巴的合理运行。如何让多个进程之间相互合作，从而实现任务的合理有序进行呢？

1.2 生产者-消费者实例

假设有一段共享数据，可以让两个进程访问和修改，这段共享数据具体来说就是上图中的结构体数组：buffer[BUFFER_SIZE]-缓存区。两个进程分别为生产者进程和消费者进程。生产者进程负责在数组中的空闲位置添加元素，消费者则从数组中的非空闲位置拿走元素。当缓存区满了的时候，生产者进程应该停下来，不能继续在里边添加元素；当缓存区空的时候，消费者进程应该停下来，不能继续从里边拿走元素。

要让两个进程合理运行，这两个进程就应该合作。生产者和消费者不可能都一直运行或一直等待，而是时而运行时而等待，进程同步就是要进程“走走停停”。

如上图所示，通过变量counter来判断缓存区的满或空，由此来决定进程是否运行或阻塞。对于生产者进程，当counter == BUFFER_SIZE时，缓存区满则停止运行，不满则counter ++，使得消费者可以使用资源；对于消费者进程，当counter == 0时，缓存区空则停止运行，不空则counter --，使得生产者可以继续生产者资源。

这里，我们使用counter作为一个信号，来决定一个进程的唤醒或睡眠，但是这样存在一些问题。

假设我们有两个生产者p1和p2，一个消费者c，此时缓存区已满，发生了以下情况：

  1. p1判断缓存区满，进入睡眠
  2. p2判断缓存区满，进入睡眠
  3. c判断缓存区非空，使用资源，counter --，判断counter == BUFFER_SIZE - 1，说明上一次缓存区满有生产者进程阻塞，唤醒了生产者p1
  4. c再次判断缓存区非空，使用资源，counter --，此时counter = BUFFER_SIZE - 2，有一个生产者进程p2还在等待，此时缓存区非满应当唤醒，但是消费者只在判断counter == BUFFER_SIZE - 1为真时，才会唤醒一个生产者就进程，因此生产者进程p2将无法唤醒

1.3 从信号到信号量

通过上面的生产者-消费者实例可以看到，仅仅通过一个信号来决定睡眠和唤醒，无法做到多进程同步，还应当记录下“有几个进程在等待”这种类似的信息。

  • 信号量：使用量来记录多进程中的一些信息，根据这些信息来发出响应的信号
  • 1965年，信号量最初是由荷兰学者Dijkstra提出的一种特殊整型变量，量用 来记录，信号用来sleep和wakeup

2 信号量临界区保护

2.1 共同修改信号量引出的问题

还是生产者-消费者问题，如上图所示，假设有两个生产者P1和P2，初始时empty = -1，在经过两个进程后，理想的情况下empty = -3，但是由于进程调度可能如上图所示，实际的结果是empty = -2，这时就导致了“竞争条件”。

  • 竞争条件：和调度有关的共享数据语义错误
  • 错误由多个进程并发操作共享数据引起
  • 错误和调度顺序有关，难于发现和调试

2.2 解决竞争条件的直观想法

  • 这种直观想法就是给共享变量上锁：在一个进程访问共享变量时阻止其他进程访问，而这段修改共享变量的代码段，我们称之为“临界区”。

  • 临界区：访问共享变量的代码片段，一定不能由多个进程同时执行

  • 解决竞争条件问题，就需要保护临界区

  • 保护临界区，就需要在进入和退出这段代码时做一些操作

  • 临界区代码的保护原则：
    • 互斥进入（基本原则）：一个进程在临界区执行时，其他进程不得进入
    • 有空让进：若干进程要求进入空闲临界区时，尽快让其中一个进程进入
    • 有限等待：进程发出进入临界区的请求，迟早都能进入临界区，不能无线等待

2.3 进入临界区的尝试

2.3.1 轮换法

  • 满足互斥进入
  • 问题：不满足有空让进。假设进程$P_0$刚执行完，此时turn = 1，该进程$P_1$执行，但是$P_1$阻塞了，临界区空闲，但是由于while (turn != 0) ;，进程$P_0$ 也无法执行临界区

2.3.2 标记法

  • 满足互斥进入

  • 满足有空等待：假设此时进程$P_0$完成，flag[0] = false，该进程$P_1$进行，但进程$P_1$阻塞，但是由于循环条件判断while (flag[1]) ;，进程$P_0$仍可进入临界区

  • 问题：无法满足有限等待

2.3.3 非对称标记法

  • 标记法使得每个进程动作都十分一致，打破这种对称可以避免标记法出现的尴尬局面

2.4 进入临界区的算法

2.4.1 Peterson算法

  • 结合了标记和轮转两种思想
  • 满足互斥进入：若两个进程都进入临界区，则flag[0] = flag[1] = true，turn == 0 == 1，矛盾
  • 满足有空让进：如果进程$P_1$不在临界区，则有flag[1] = flase或者turn = 0，进程$P_0$可以进入临界区
  • 满足有限等待：如果进程$P_0$要求进入临界区，而此时进程$P_1$正在执行，则在$P_1$执行后，flag[1] = false，进程$P_0$便会得到执行

2.4.2 面包店算法

  • 多个进程，结合了标记和轮转的结合

  • 轮转：每个进程都获得一个序号，序号最小的进入

  • 标记：进程离开时序号为0，不为0的序号即标记

  • 满足互斥进入

  • 满足有空让进：当临界区空闲时，序号最小的进程一定能够进入

  • 满足有限等待：来开临界区的进程再次请求进入时，重新分配的序号一定是最大的，所有进程符合先进先出的规则

2.5 临界区保护的另一类解法

  • 临界区只允许一个进程进入，防止其他进程进入
  • 只有其他进程不被调度就不会进入临界区，进程不被调度就可以停止中断

  • 上如这种方法，只在单核CPU下有效，多核时仍然存在问题

  • 通过一些硬件的方法，让进程在临界区执行时，以原子方式执行

  • 原子方式：指令要么没运行，要么运行完成，没有中间状态

3 死锁处理

我们仍然看生产者-消费者实例

如上图所示，假如生产者和消费者函数中的P()语句前后调换，初始时，mutex = 1，empty = 0。生产者进程执行，执行P (mutex)后mutex = 0，执行P (empty)后empty = -1，睡眠进程；消费者进程执行，执行P (mutex)后mutex = -1，进程睡眠。此时，要执行生产者程序，就需要消费者继续执行V ()函数，者取决于mutex = 0，而修改mutex又取决于生产者自身，这就发生了“死锁”

  • 死锁：多个进程由于互相等待对方持有的资源而造成的谁都无法执行的情况

3.1 死锁分析

  • 死锁的成因：
    • 资源互斥使用，一旦占有别人无法使用
    • 进程占有了一些资源，又不释放，再去申请其他资源
    • 各自占有的资源和互相申请的资源形成了环路等待
  • 死锁的4个必要条件
    • 互斥使用
    • 不可抢占：资源只能等别的进程自愿放弃
    • 请求和保持：占有资源后再去申请其他资源
    • 循环等待
  • 死锁处理方法概述：
    • 死锁预防：破坏死锁出现的条件
    • 死锁避免：检测每个资源请求，如果造成死锁就拒绝
    • 死锁检测+恢复：检测到死锁出现时，让一些进程回滚，让出资源
    • 死锁忽略：忽略出现的死锁

  • 在进程执行前，一次性申请所有需要的资源

两种方法都会造成资源浪费
